# AGENTS.md

## 프로젝트 목표
- 초보자도 어렵지 않게 주식에 접근할 수 있는 웹사이트를 만드는 것입니다.
- 복잡한 시장 지표/차트를 최소한의 용어로 설명하고, 직관적 기능 흐름(검색, 관심종목, 과거사례, 모의투자)으로 학습 장벽을 낮추는 것을 최우선으로 둡니다.
- 기술적 정확성은 유지하되, 학습 흐름이 먼저 오도록 UI/UX와 피드백을 설계합니다.

## 핵심 사용자 가설
- 사용자는 종목명, 상식 수준의 검색어, 투자 개념을 잘 모르는 상태에서 시작한다.
- 즉시 이해 가능한 텍스트와 단계형 가이드를 통해 행동(조회/비교/매수 시뮬레이션)을 수행할 수 있어야 한다.
- 실패/오류/데이터 지연이 나더라도 “왜 이런 결과인지”를 설명해주는 피드백이 중요하다.

## 역할 정의
- 사용자: 프로젝트 방향성 결정, 우선순위 지정, 기획/요구사항 확정
- Codex: 설계 제안, 구현, 디버깅, 검증, 결과 정리

## 현재 우선순위 (현재 비워둠)
1. 회원가입/로그인/오류 대응 플로우 안정화 (백엔드 로그 기반 디버깅 루틴 정리).
2. 초보자용 모의 투자 시뮬레이션 완성: 주문-평단-손익 계산, 수수료 반영, 실패 사유 설명.
3. 학습 플로우 강화: 검색 → 종목 이해 → 모의 주문의 단일 스텝 가이드 구현.

## 앞으로 해야 할 작업
- 모의 투자 UI/UX 점검
  - 초보자용 단계형 안내(매수/매도 흐름) 추가
  - 주문 실패 메시지 개선(현금부족/미보유수량/입력오류 시 원인 설명)
- 포트폴리오/시뮬레이션 데이터 일관성
  - 백엔드 보유종목과 데모 계좌 현금 상태 동기화 전략 정리
  - 필요 시 데모 전용 계좌 상태 저장 API 도입 여부 결정
- 테스트/검증
  - 로그인·회원가입·포트폴리오 API 연동 로그 체크리스트 작성
  - 프론트 핵심 기능(시뮬레이션 주문/갱신/초기화) 수동 테스트 케이스 정리
  

## 실행 규칙
- 모든 빌드/테스트/의존성 설치는 컨테이너 내부에서 수행한다.  
  `docker-compose` 또는 `docker build/run` 같은 컨테이너 진입 경로를 기준으로 작업 결과를 검증한다.
- 백엔드/프론트 핵심 변경 시 영향 범위를 함께 정리한다.
- 기능 구현 전 사용자 확인이 필요한 모호한 부분은 명확한 질문 1개로 좁혀서 진행한다.
- 로그 기반 디버깅이 필요한 경우, 재현 단계 + 해당 로그 캡처 방법을 먼저 기록한다.
- 프론트엔드 변경 후 확인은 `frontend/FRONTEND_QUICKCHECK.md` 기준으로 즉시 수행한다.
- 사용자 승인 없는 `회의 기록`/약속 기록은 `회의 기록`에 즉시 추가하지 않는다.
- 테스트 강제 적용(필수)
  - 기능 개발 단위마다 관련 백엔드 API/로직과 프론트 UI 경로의 단위 테스트를 함께 작성한다.
  - 구현 완료 후 관련 테스트를 실행하지 않으면 변경 완료를 승인하지 않는다.
  - 기본 실행 규칙
    - 백엔드: `pytest` 또는 `pytest -k <target>`로 변경 모듈을 대상으로 실행
    - 프론트: `npm run test:run -- --run tests/<target>.test.*` 또는 `npm run test:run`으로 실행
  - 예외는 사용자 동의가 있을 때만 허용한다.
